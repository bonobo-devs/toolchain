# Bonobo Userscript Distribution Format v1 (BUDFv1) specifications
## Preamble
The BUDFv1 file format is an initial attempt at creating an open standard for the communication and storage of userscripts and their metadata. 

Because the metadata fields are encoded in standard JSON, most programming languages can easily interpret BUDFv1 files without the need for external libraries.

## Definitions
The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED",  "MAY", and "OPTIONAL" in this document are to be interpreted as described in [RFC 2119](https://datatracker.ietf.org/doc/html/rfc2119).

## Structure of a BUDFv1 file
A BUDFv1-compliant file contains a structure similar to the codeblock below: (though the order of each key and the general nature of the file's formatting MAY differ, and this standard applies no preference to any given formatting decisions):
~~~json
{
   "BUDFVersion":1,
   "version":"v0.1.0",
   "name":"My Awesome Userscript",
   "id":"my-awesome-userscript",
   "match":[
      "*://*/*"
   ],
   "code":"alert(\"Hello world!\");console.warn(\"This is a warning.\");"
}
~~~

### Keys

This is a list of keys that can be included in a BUDFv1-compliant file as well as directions on their proper usage and application.

Keys marked as "required" MUST be included in all BUDFv1-compliant files.

#### `BUDFVersion` key
In BUDFv1-compliant files, this required key's value MUST be equal to the integer 1, indicating the applicable standard version.

Applications that parse the contents of BUDFv1-compliant files SHOULD implement a check to make sure that the `BUDFVersion` key is equal to 1, and produce an error or warning upon receiving a non-compliant file, as there may be scenarios in which a future version of the BUDF standard could cause compatibility issues with older software.

#### `version` key
In BUDFv1-compliant files, this required key's value MUST contain a human-readable string representing the version of the userscript.

Applications that comply with the BUDFv1 standard SHALL NOT rely upon this value as a method of judging which of two BUDFv1-compliant files contains a newer or older version of a userscript.

#### `name` key
In BUDFv1-compliant files, this required key's value MUST contain a human-readable string indicating the name of the userscript. For example, this key may be set to `"My Awesome Userscript"` to indicate that the name of the userscript is "My Awesome Userscript".

BUDFv1-compliant applications MUST NOT use this key's value as the sole method of determining whether or not two BUDFv1-compliant files contain the same or similar contents. Compliant applications MAY produce an error or warning message upon being instructed to install a BUDFv1-compliant userscript that shares its `name` value with an already-installed userscript.

#### `id` key
In BUDFv1-compliant files, this required key's value MUST contain a single-line, all-lowercase string without any spaces or non-ASCII characters for the purposes of providing a machine-readable ID to permit compliant applications to properly sort and store version(s) of a userscript, as seen fit by said applications.

This key's value SHOULD be different from the value of the `name` key.

#### `match` key
In BUDFv1-compliant files, this required key's value MUST contain an array of strings that are valid [match patterns](https://developer.chrome.com/docs/extensions/develop/concepts/match-patterns).

Whenever a page's URL matches one of these patterns, a compliant browser extension MUST execute the corresponding code (see the `code` key's documentation below), unless the user has requested otherwise or browser security measures prevent such code execution from occurring.

The aforementioned array SHOULD contain at least one item and MUST contain no items with non-string data types.

#### `code` key
In BUDFv1-compliant files, this required key's value MUST contain a string of valid ECMAScript (JavaScript) code to be executed in the browser's `MAIN` world whenever one of the match patterns from the `match` key (see above) is matched.
